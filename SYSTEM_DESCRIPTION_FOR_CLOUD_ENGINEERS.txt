================================================================================
SYSTEM OVERVIEW
================================================================================

JR Fleet Management System is a comprehensive fleet operations platform designed
for multi-country vehicle fleet management across Ghana, Liberia, and São Tomé
and Príncé. The system handles vehicle tracking, maintenance scheduling, driver
management, fuel tracking, inventory management, safety compliance, and financial
reporting with multi-currency support.

This is a pilot deployment for GTI Holdings to validate the solution before
scaling to full production.

TECHNICAL ARCHITECTURE
================================================================================

APPLICATION STACK:
------------------
Backend:
- Framework: FastAPI (Python 3.11+)
- API Design: RESTful with /api prefix
- Authentication: JWT-based with role-based access control (RBAC)
- API Documentation: OpenAPI/Swagger auto-generated
- Server File: backend/server.py (2,617 lines, 100+ endpoints)

Frontend:
- Framework: React 19.0 with React Router DOM 7.5.1
- UI Library: Radix UI components with Tailwind CSS
- State Management: React Context API (AuthContext)
- Build Tool: Create React App (CRA) with CRACO configuration
- Pages: 25+ dashboard pages (Dashboard, Vehicles, Drivers, Maintenance, Reports, etc.)

Database:
- Type: MongoDB-compatible (document-oriented NoSQL)
- Schema: 20+ collections with complex relationships
- Key Collections: vehicles, drivers, maintenance_records, fuel_transactions,
  expenditures, users, alerts, compliance_checks, inventory, documents
- Multi-tenancy: Country-scoped data (GHANA | LIBERIA | SAO_TOME)

PERFORMANCE & CAPACITY EXPECTATIONS
================================================================================

Pilot Phase Targets:
- Concurrent Users: 700

- Database Connections: ~150 concurrent

Scaling Considerations:

- Database read-heavy workload (80% reads, 20% writes)
- Potential for read replicas in production phase
- Session storage can use database temporarily (no Redis required for pilot)

Known Performance Characteristics:
- AI predictions: 2-5 seconds per request (Google Gemini API latency)
- Document uploads: File size limit 10 MB per document
- Bulk operations: Some endpoints support batch processing

Data Protection:

- Role-based access control (5 role levels)

- Document retention policies

- Private subnets for backend compute and database
- Public subnets for load balancer only
- Security groups with least-privilege rules
- No direct internet access from compute layer

Secret Management:
- Environment variables should use secrets management service
- API keys rotation capability required

Rate Limiting:
- Consideration for API endpoint protection
- Authentication endpoint brute-force protection

================================================================================
CONSTRAINTS & LIMITATIONS
================================================================================

Pilot Phase Acceptable Limitations:
- Single availability zone deployment acceptable (no multi-AZ)
- Spot instance interruptions acceptable (2-minute recovery window)
- Basic monitoring (critical alarms only)
- No advanced caching layer (document database caching sufficient)
- Minimal WAF protection (basic rate limiting acceptable)
- 3-7 day log retention (not long-term archival)

Not Required for Pilot:
- High availability (HA) database clustering
- Disaster recovery (DR) site
- Advanced DDoS protection
- Real-time analytics dashboard
- Automated scaling beyond basic thresholds
- Multi-region deployment

Future Production Requirements:
- Multi-AZ database with automatic failover
- Read replicas for performance
- Redis caching layer
- Advanced WAF rules
- Extended log retention (30+ days)
- Backup retention >7 days
- Performance optimization (sub-500ms API responses)
